## 4. SCHEDULING — COMPLETION DATE MUST BE AT LEAST 1 DAY BEFORE AVAIL DATE (AUTO-ADJUST)

Implement an automatic scheduling adjustment to ensure that the Completion Date of an order does not go beyond its Avail Date. The Completion Date must be at least 1 day (24 hours) before the Avail Date.

**Auto-adjust logic:**

**Step 1 — Calculate the natural Completion Date:**
- Completion Date = Start Date/Time + Production Hours + Changeover Time

**Step 2 — Compare against Avail Date:**
- If the Avail Date is a valid date (not "prio replenish", "safety stocks", "for sched", or any non-date text):
  - Calculate the deadline: Avail Date minus 1 day (24 hours). This is the latest acceptable Completion Date.
  - If the natural Completion Date is ON or BEFORE this deadline → No adjustment needed. Keep the natural schedule.
  - If the natural Completion Date is AFTER this deadline → Auto-adjust is required.

**Step 3 — Auto-adjust the Start Date:**
- Work backwards from the deadline:
  - Required Completion Date = Avail Date - 1 day (24 hours)
  - Required Start Date/Time = Required Completion Date - Production Hours - Changeover Time
- Set the order's Start Date and Start Time to the calculated required values.
- If the required Start Date is earlier than the current date, push it to the previous day(s) as needed. There is no restriction on how far back it can go.

**Step 4 — Check for upstream conflicts:**
- After adjusting the Start Date, check if the new Start Date/Time overlaps with (is earlier than) the previous order's Completion Date in the same line.
- If there IS an overlap (the adjusted order would need to start before the previous order finishes):
  - Flag the row with a red highlight (light red background #ffebee) indicating a scheduling conflict.
  - Display a red warning icon (⚠) in the Start Date cell with tooltip: "Scheduling conflict: This order's adjusted start time overlaps with the previous order's completion. Resolve by reordering or adjusting upstream orders."
  - The auto-adjusted Start Date is still applied — the system shows where the order NEEDS to start, even if it conflicts. The user must resolve the conflict manually (reorder, adjust upstream, etc.).

**Step 5 — Cascade downstream:**
- After auto-adjusting, recascade all downstream orders:
  - Next order's Start Date/Time = This order's Completion Date + Changeover Time
  - Continue cascading down the line.
- If downstream orders also have Avail Dates, apply the same auto-adjust logic to them recursively.

**Orders excluded from auto-adjust:**
- Orders with non-date Avail Date values ("prio replenish", "safety stocks", "for sched", "ASAP", or any free-text) are EXCLUDED from auto-adjust.
- These orders do not have a deadline to meet. Their scheduling is determined purely by cascade (previous order's Completion Date + Changeover Time) or by manual user placement.
- Users intentionally place these orders in gaps between dated orders to maximize production uptime.

**Gap utilization for non-dated orders:**
- When two dated orders have a time gap between them (e.g., Order 1 completes at 12:00 PM and Order 2 must start at 3:00 PM to meet its Avail Date), there is a 3-hour gap.
- Non-dated orders ("prio replenish", "safety stocks") can be manually dragged into this gap by the user.
- When a non-dated order is placed between two dated orders:
  - Its Start Date/Time = Previous order's Completion Date + Changeover Time
  - Its Completion Date = Start Date/Time + Production Hours + Changeover Time
  - If the non-dated order's Completion Date would push into the next dated order's Start Date, flag the non-dated order with a yellow warning (#fff9c4 background): "This order may not fit in the available gap. Completion at [time] overlaps with next order starting at [time]."

**Conflict highlighting:**

| Conflict Type | Row Background | Icon |
|---|---|---|
| Upstream overlap (adjusted start before previous completion) | Light Red #ffebee | Red ⚠ in Start Date cell |
| Non-dated order exceeds gap | Light Yellow #fff9c4 | Yellow ⚠ in Completion Date cell |
| Completion Date exceeds Avail Date (if auto-adjust fails) | Light Red #ffebee | Red ⚠ in Completion Date cell |

**First order in a line (Prio 1):**
- If the first order needs to start earlier than 8:00 AM on the upload date to meet its Avail Date constraint, the system pushes the Start Date to the previous day(s) as needed.
