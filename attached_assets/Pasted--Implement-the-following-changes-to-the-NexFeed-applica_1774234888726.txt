## 4. "DONE" STATUS — BLANK PRIO AND RENUMBER ACTIVE ORDERS

When an order's status is set to "Done", its Prio number becomes blank, and all remaining active orders renumber starting from 1.

### BEHAVIOR WHEN ORDER IS SET TO "DONE"

1. The Done order moves to the top of the table (existing behavior).
2. The Done order's Prio column becomes **BLANK** — empty cell, no number.
3. All remaining ACTIVE orders are **renumbered starting from Prio 1**.

### EXAMPLE — SINGLE ORDER DONE

**Before:**
```
Prio 1: Order A | Plotted
Prio 2: Order B | In production
Prio 3: Order C | Plotted        ← user sets to "Done"
Prio 4: Order D | Plotted
Prio 5: Order E | Plotted
```

**After:**
```
(blank): Order C | Done           ← moved to top, Prio BLANK
Prio 1:  Order A | Plotted        ← renumbered from 1
Prio 2:  Order B | In production
Prio 3:  Order D | Plotted
Prio 4:  Order E | Plotted
```

### EXAMPLE — MULTIPLE DONE ORDERS

```
(blank): Order F | Done           ← most recent Done, blank Prio
(blank): Order C | Done           ← previous Done, blank Prio
Prio 1:  Order A | Plotted        ← first active order
Prio 2:  Order B | In production
Prio 3:  Order D | Plotted
Prio 4:  Order E | Plotted
```

### CANCELLED ORDERS — ALSO BLANK PRIO

Cancelled orders at the bottom also have blank Prio:

```
(blank): Order F | Done           ← top, blank
(blank): Order C | Done           ← top, blank
Prio 1:  Order A | Plotted
Prio 2:  Order B | In production
Prio 3:  Order D | Plotted
(blank): Order G | Cancel PO      ← bottom, blank
```

### HOLD ORDERS — KEEP PRIO NUMBER

Orders on "Hold" KEEP their Prio number because they are still active (just paused):

```
(blank): Order F | Done
Prio 1:  Order A | Plotted
Prio 2:  Order B | Hold           ← keeps Prio 2, greyed out
Prio 3:  Order D | Plotted
Prio 4:  Order E | Plotted
```

### PRIO DISPLAY RULES

| Status | Prio Display |
|---|---|
| Done | Blank (empty cell) |
| Cancel PO | Blank (empty cell) |
| Hold | Numbered (keeps position) |
| Plotted | Numbered |
| Cut | Numbered |
| In production | Numbered |
| On-going batching | Numbered |
| On-going pelleting | Numbered |
| On-going bagging | Numbered |
| Combined with other PO | Numbered |

### REVERTING FROM "DONE" BACK TO ACTIVE

When a user changes an order's status FROM "Done" back to any active status:

1. The order moves FROM the Done section at the top back to its **previous position** in the active section.
2. The order receives its previous Prio number (or the nearest available if positions shifted).
3. All active orders renumber to accommodate.

**Example — Revert:**

**Current (Order C is Done):**
```
(blank): Order C | Done
Prio 1:  Order A | Plotted
Prio 2:  Order B | In production
Prio 3:  Order D | Plotted
```

**User reverts Order C from Done to Plotted (previously was Prio 3):**
```
Prio 1:  Order A | Plotted
Prio 2:  Order B | In production
Prio 3:  Order C | Plotted        ← returned to previous position
Prio 4:  Order D | Plotted
```

**Implementation:**
- Store the order's previous Prio position when it's set to Done.
- Field: `previous_prio` — set when status changes to Done, cleared when status changes away from Done.
- On revert: place the order back at `previous_prio` position (or at the end of active orders if the position no longer exists).

### RENUMBERING TRIGGER

Prio renumbering runs after ANY of these events:
- Order set to Done
- Order set to Cancel PO
- Order reverted from Done to active
- Order reverted from Cancel PO to active
- Drag-and-drop reorder
- Combine / Uncombine
- Cut / Merge back
- Auto-Sequence applied
- Any order addition or deletion

- The ENTIRE block (lead + all children) moves to the top.
- ALL orders in the block get blank Prio.
- Active orders below renumber from 1.

(blank): [Child] Order B-2 | Done   ← blank
