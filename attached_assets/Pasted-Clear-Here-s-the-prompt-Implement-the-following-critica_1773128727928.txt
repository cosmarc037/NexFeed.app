## FIX: SCHEDULING ANCHOR — ENTIRE TIMELINE WORKS BACKWARDS FROM AVAIL DATE

The current scheduling logic is fundamentally broken. It starts the timeline from the upload date (current date) and cascades FORWARD, ignoring the Avail Date entirely. This results in orders completing days or weeks before they need to, leaving the plant with no work to do after the orders finish.

**The correct logic:** The ENTIRE timeline must be anchored to the Avail Date and work BACKWARDS. The Avail Date is the fixed endpoint. Everything else is calculated backwards from it.

---

### STEP-BY-STEP SCHEDULING ALGORITHM

**Step 1 — Identify the anchor:**
- Find all orders in the line with valid Avail Dates.
- Sort them by Avail Date, EARLIEST first.
- If multiple dated orders exist, each one is an anchor point.
- If only one dated order exists, that is the single anchor.
- If NO dated orders exist (all orders are prio replenish / safety stocks), fall back to: first order starts at upload date 08:00 AM, cascade forward. STOP — skip remaining steps.

**Step 2 — Position the LAST dated order:**
- The last dated order (latest Avail Date) must complete ON or BEFORE its Avail Date.
- Work backwards:
  - Completion Date/Time = Avail Date, reasonable production hour (e.g., 06:00 PM, or end of production day, or simply the Avail Date itself — the order must NOT complete AFTER the Avail Date)
  - Start Date/Time = Completion Date/Time - Production Hours

**Step 3 — Position earlier dated orders:**
- For each dated order (working from latest Avail Date backwards to earliest):
  - Each dated order must complete ON or BEFORE its own Avail Date.
  - Work backwards from its Avail Date to determine its Start Date/Time.

**Step 4 — Identify gaps between dated orders:**
- Calculate the time gaps between consecutive dated orders:
  - Gap = Next dated order's Start Date/Time - Previous dated order's Completion Date/Time
- Calculate the time gap BEFORE the first dated order:
  - Gap = First dated order's Start Date/Time - (whatever time is available before it)

**Step 5 — Fill gaps with non-dated orders (BACKWARDS from each anchor):**
- Collect all non-dated orders (prio replenish, safety stocks, etc.).
- Starting from the gap immediately BEFORE the first dated order:
  - Stack non-dated orders backwards from the first dated order's Start Date/Time.
  - Last non-dated order's Completion = First dated order's Start Date/Time - Changeover
  - Second-to-last non-dated order's Completion = Last non-dated order's Start - Changeover
  - Continue stacking backwards until all non-dated orders that fit are placed, or we reach the upload date.
- If there are gaps between dated orders, fill those with remaining non-dated orders using the same backwards stacking.
- Any non-dated orders that still don't fit go after the last dated order.

**Step 6 — Determine the FIRST order's Start Date/Time:**
- After stacking everything backwards, the FIRST order (Prio 1) in the line gets its Start Date/Time as the result of the backwards calculation.
- This could be March 21 at 6:00 PM, or March 20 at 2:00 AM, or any date — it is whatever the math produces.
- The first order does NOT default to the upload date. It defaults to whatever the backwards calculation produces.

**Step 7 — Final forward cascade verification:**
- After the backwards calculation determines all positions, run a forward cascade from Prio 1 to verify:
  - Order 1: Start = calculated, Completion = Start + Production Hours
  - Order 2: Start = Order 1 Completion + Changeover, Completion = Start + Production Hours
  - ... continue for all orders
  - Last dated order: Verify Completion is ON or BEFORE Avail Date. ✅

- Must complete on or before March 22, 2026.

### EDGE CASES

**Edge case 1 — Non-dated orders exceed available time before first anchor:**
- If non-dated orders require MORE time than available before the first dated order:
  - Fill as many non-dated orders as possible before the first dated order.
  - Place remaining non-dated orders in gaps between dated orders.
  - If gaps are also full, place remaining after the last dated order.

**Edge case 2 — Dated order Avail Date is today or tomorrow:**
- Work backwards as normal.
- If the resulting first order Start Date is in the past or earlier today:
  - Set the first order Start to NOW (current time) or 08:00 AM today, whichever is later.
  - Cascade forward.
  - If the dated order's Completion exceeds its Avail Date, flag as conflict (red highlight).

**Edge case 3 — All orders are non-dated:**
- No anchor exists.
- First order starts at upload date, 08:00 AM.
- Cascade forward continuously.

**Edge case 4 — Avail Date is in the past:**
- Flag as overdue (red highlight).
- Treat as highest priority — place at the beginning of the timeline.
- Start immediately (upload date, 08:00 AM or current time).

**Edge case 5 — Multiple dated orders with the same Avail Date:**
- Sort by Volume (larger first) or maintain upload order.
- Both must complete by the same Avail Date.
- Stack them sequentially, both completing before the Avail Date.

---

### IMPORTANT — WHAT THIS REPLACES

This REPLACES the previous rules:
- ❌ "First order defaults to upload date, 08:00 AM" — NO. First order's Start Date is calculated by the backwards algorithm.
- ❌ "Cascade forward from current date" — NO. The cascade is verified forward, but the POSITIONS are determined by working backwards from the Avail Date anchor(s).
- ❌ "Auto-adjust only when there's a conflict" — NO. The entire timeline is ALWAYS anchored to Avail Dates from the start. There should never be a scenario where a dated order completes 12 days early.

The upload date (current date) is only used as the Start Date when ALL orders are non-dated.
