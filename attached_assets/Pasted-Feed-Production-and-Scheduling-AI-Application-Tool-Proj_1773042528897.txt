Feed Production and Scheduling AI Application Tool
Project Nuevo | Detailed Prompt
---

Create an AI-assisted platform ("NexFeed") to support production teams by automating the extraction and processing of feed orders from SAP-exported Excel files. The platform should interpret and parse order data from these files and update the production schedule efficiently. By minimizing manual data entry, the system should streamline scheduling, reduce errors, and enhance visibility into production status, ultimately improving workflow efficiency and enabling better decision-making in feed manufacturing operations.

============================================================================================
PRIMARY OBJECTIVE
============================================================================================

The client currently receives feed order details exported from SAP as Excel files. They manually extract and transfer this data into separate Excel-based production schedules, resulting in a time-consuming and error-prone process that hinders efficient order tracking and production management. 

The goal is to develop an application that automates this workflow by allowing users to upload SAP-exported Excel files, automatically parse relevant data from each cell, and update the production schedule within the system seamlessly. This automation should significantly reduce manual effort, increase data accuracy, and provide a unified, real-time view of production schedules to support operational efficiency and informed decision-making.

============================================================================================
TARGET AUDIENCE
============================================================================================

The application will serve the following stakeholders involved in feed production
scheduling:

•	Production Planners and Schedulers: Manage daily feed production schedules and require a streamlined, automated solution to replace manual data transfers from SAP exports.
•	Operations Managers: Oversee production efficiency and resource allocation; need accurate, real-time production schedule visibility to make effective decisions.
•	IT and Data Support Teams: Support production processes and aim to minimize errors from manual data handling and simplify SAP data integration.
•	Feed Manufacturing Companies: Utilize SAP for order planning and seek to improve operational efficiency by automating production tracking and scheduling.

============================================================================================
TARGET AVAIL DATE — PARSING RULES
============================================================================================

The Remarks column in the SAP Excel file may contain date information in the following formats:

Sample Remarks values:
•	"TLD | Jan 03"
•	"TLD | Jan 06"
•	"prio replenish"
•	“safety stocks”

Parsing behavior:

1.	If the Remarks value contains a recognizable date pattern (e.g., "TLD | Jan 03"), extract the date portion ("Jan 03") and infer the year based on context (current or next calendar year). Store the parsed date in the Avail Date column.
2.	If the Remarks value contains non-date text (e.g., "prio replenish"), display it as-is in the Avail Date field. Do not auto-convert it to a date or label it as "undetermined." Show the original text value.
3.	Non-date values in Avail Date (such as "prio replenish") should trigger a red X readiness status, since they are not valid production dates.
4.	Users can override the Avail Date field using the override mechanism (lock icon + "Confirm" dialog) to replace non-date text with a proper date. Once overridden, the value is stored as a proper date and the readiness status recalculates accordingly.
5.	This means the Avail Date field supports two states:
1.	Text mode: Displays the raw parsed text from Remarks (read-only display with override mechanism available).
2.	Date mode: Once a proper date is set (either parsed or manually overridden), it displays as a formatted date (MMMM DD, YYYY).

Avail Date — Display behavior:

•	When the SAP file is uploaded and the Remarks column contains non-date text (e.g., "prio replenish", "rush order", or any other text), the raw text must be displayed as-is in the Avail Date field. Do NOT hide, omit, or replace it.
•	Display behavior for non-date values:
Line 1: prio replenish (displayed as-is, raw text from Remarks)
Line 2: (blank — original Remarks reference if applicable, smaller grey text)
•	The field should clearly show the original text so users understand the context of the order.
•	Non-date text values still trigger a red X readiness status since they are not valid production dates.
•	When the user clicks the lock icon to override, the confirmation dialog appears, and upon confirmation the calendar picker opens allowing them to set a proper date.
•	Once a proper date is selected, the text is replaced with the formatted date (MMMM DD, YYYY) and the readiness status recalculates.
•	The Start Date and Start Time remain blank when Avail Date is non-date text. Once a valid date is set, the AI suggests Start Date/Time based on the new deadline.

============================================================================================
CALCULATION RULES
============================================================================================

The following fields are auto-calculated based on available data:

1.	Suggested Volume (MT):
•	Take the Original Volume from the SAP file (Metric Ton column).
•	If the Original Volume is exactly divisible by the Batch Size, the Suggested Volume equals the Original Volume.
•	If the Original Volume is NOT divisible by the Batch Size, round UP to the nearest multiple of the Batch Size.
•	Formula: Suggested Volume = ceil(Original Volume / Batch Size) * Batch Size
•	Examples:
- Original: 19 MT, Batch Size: 4 → Suggested: 20 MT
- Original: 6 MT, Batch Size: 4 → Suggested: 8 MT
- Original: 8 MT, Batch Size: 4 → Suggested: 8 MT
- Original: 3 MT, Batch Size: 2.5 → Suggested: 5 MT
- Original: 7 MT, Batch Size: 3 → Suggested: 9 MT
•	The Suggested Volume is used for ALL downstream calculations: Number of Batches, Number of Bags, Production Hours, and capacity tracking.
•	Both Original Volume and Suggested Volume are non-editable.

2.	Number of Batches:
Formula: Suggested Volume (MT) / Batch Size
Example: 20 MT / 4 = 5 batches

3.	Number of Bags:
Formula: Suggested Volume (MT) / 50 * 1000
Example: 20 MT / 50 * 1000 = 400 bags

4.	Recalculation behavior:
•	Both Original Volume and Batch Size are non-editable. They are sourced from the SAP file and Knowledge Base respectively.
•	Number of Batches and Number of Bags are calculated outputs.
•	Recalculation occurs when the Knowledge Base is re-applied (which may change Batch Size), causing Suggested Volume to recalculate, which cascades to all dependent fields.

5.	Changeover Time:
Default value: 0.17
Non-editable.
Note: In future iterations, the changeover time may vary per "Form" column value. For now, 0.17 is used as a fixed default.

6.	Production Hours:
Formula: Suggested Volume (MT) / Run Rate
Always round to 2 decimal places.

Run Rate source:
•	The application references a Knowledge Base that contains run rates per line number.
•	Run Rate is auto-populated based on the order's specific line number:
- Line 1 → KB "Line 1 Run Rate"
- Line 2 → KB "Line 2 Run Rate"
- Line 3 → KB "Line 3 Run Rate"
- Line 4 → KB "Line 4 Run Rate"
- Line 5 → KB "Line 5 Run Rate"
- Line 6 → KB "Line 6 Run Rate"
- Line 7 → KB "Line 7 Run Rate"
•	If Form = "M" (Mash), there is NO run rate — Production Hours cannot be calculated and remains BLANK.
•	If any other Form has no matching Run Rate in the Knowledge Base, Production Hours also remains BLANK.
•	Run Rate and Production Hours are non-editable.

7.	Completion Date:
Formula:
•	If Production Hours has a value:
Completion Date = Start Date/Time + Production Hours + Changeover Time
•	If Production Hours is BLANK (no run rate):
Completion Date = Start Date/Time + Changeover Time
•	If any of Start Date, Start Time, Production Hours, or Changeover Time are blank, Completion Date displays as blank.
Non-editable (calculated output). Recalculates automatically when inputs change.

8.	Scheduling Cascade Logic:
Production operates 24 hours continuously — no shift breaks, no downtime windows. All time calculations are continuous.

Order sequencing within a feedmill line:
•	Order 1 (first row in the uploaded file) becomes the default starting order unless the user reorders via drag-and-drop.
•	Order 1 Start Date/Time: User-defined or system default.
•	All subsequent orders: Start Date/Time = Previous order's Completion Date.

Cascade rules:
•	When any order's values change (Start Date/Time, Production Hours, Suggested Volume, or row position), recalculate that order's Completion Date, then cascade to ALL subsequent orders within the SAME feedmill line.
•	Changes in one feedmill line do NOT affect other feedmill lines.

Cascade trigger events:
•	First upload — calculate all orders in sequence per feedmill line.
•	Drag-and-drop reorder — recalculate all affected orders in the reordered feedmill line.
•	Manual edit of Start Date/Time — recalculate that order's Completion Date, then cascade downstream.
•	Knowledge Base re-application — may change Batch Size and Run Rate, causing Suggested Volume and Production Hours to recalculate, then cascade.

Summary calculation flow per order in sequence:
1.	Start Date/Time =
- If Order 1 (first row): User-defined / system default
- If Order N: Previous order's Completion Date
2.	Production Hours =
- If Run Rate exists: Suggested Volume (MT) / Run Rate → round to 2 decimal places
- If Form = "M" or no Run Rate: BLANK
3.	Changeover Time = 0.17 hours (fixed default)
4.	Completion Date =
- If Production Hours exists: Start Date/Time + Production Hours + 0.17
- If Production Hours is BLANK: Start Date/Time + 0.17
5.	Cascade to next order in the same feedmill line...

============================================================================================
DEFAULT SORT ORDER
============================================================================================

First Upload (no existing orders in the app):
•	Orders are sorted by Avail Date in ascending order (earliest date first).
•	Orders with valid dates come FIRST, sorted chronologically.
•	Orders with non-date text in Avail Date (e.g., "prio replenish", "safety stock") come AFTER all dated orders.
•	Among non-date text orders, preserve the original row order from the Excel file.
•	Within groups (same date or non-date), original Excel row order is preserved.

Subsequent Uploads (orders already exist):
•	Existing orders are NOT re-sorted. Their positions are preserved exactly as-is.
•	New orders are APPENDED BELOW all existing orders as a block.
•	Within the new block, the same sorting rules apply: valid dates first (ascending), then non-date text orders.
•	This append behavior applies PER feedmill line.

Post-upload:
•	Users can manually reorder via drag and drop.
•	Users can sort by clickable column headers.
•	No automatic re-sorting after initial upload sort.

--------------------------------------------------------------------
COMBINED ORDERS — ROW BEHAVIOR
--------------------------------------------------------------------

When orders have Status = "Combine with other PO":

Visual distinction:
•	All combined orders in the same group are highlighted with a light blue tint (#e0f2f8) as the row background color.
•	The lead order (highest Suggested Volume in the group) has a small "Lead" badge/indicator next to its FPR.

Sticky grouping:
•	Combined orders in the same group are always positioned adjacent to each other in the table.
•	When the lead order is dragged to a new position, all combined orders in the group move together.
•	Individual combined orders CANNOT be dragged separately. To separate an order from the group, the user must first change its status away from "Combine with other PO."

Status sync:
•	When the lead order's status is changed (e.g., to "In production", "Hold", "Done", "Cancel PO"), ALL combined orders in the group automatically update to the same status.
•	When a NON-LEAD order's status is changed independently, that order BREAKS OUT of the combine group:
-	The breakout order becomes independent with the newly selected status.
-	FPR Notes referencing the combine are removed from the breakout order.
-	The remaining combined orders stay grouped.
-	If the breakout order was NOT the lead, the lead remains the same.
-	If only 2 orders were combined and one breaks out, the remaining order also becomes independent (no group of 1). FPR Notes referencing the combine are removed from both.
•	If the LEAD order breaks out, the next highest Suggested Volume order in the remaining group becomes the new lead.

Shared fields for combined orders:
•	Status, Start Date, Start Time, and End Date are synchronized across all orders in a combine group (changes to the lead propagate to all).
•	HA Info, HA Prep, FPR Notes, and Special Remarks are editable independently per order within the group.

Capacity tracking for combined orders:
•	The total combined Suggested Volume (MT) of all orders in the group counts toward line capacity when the group status is "In production" or any "On-going *" status.

--------------------------------------------------------------------
TOP 3 COMPLETED ORDERS DISPLAY
--------------------------------------------------------------------

At the top of the main interactive order table in each feedmill line tab, up to 3 recently completed orders (Status = "Done") are displayed in a fixed section.

Behavior:
•	Completed orders are sorted with the oldest "Done" first (chronological order — the first order completed is row 1).
•	These rows are highlighted with a light green tint (#e8f5e9) to visually distinguish them from active orders.
•	These rows are FIXED — they cannot be dragged or reordered.
•	All fields in completed order rows are read-only (Status dropdown remains editable for status changes such as reverting to In production).
•	A thin visual separator (horizontal line or subtle divider) separates the completed orders section from the active orders below.

Maximum of 3 completed orders displayed:
•	When a 4th order is marked as "Done," the oldest completed order (row 1 of the completed section) is automatically moved to the Order History in the Configurations tab.
•	The newly completed order takes the 3rd position in the completed section.
•	This rotation happens automatically upon any order being marked "Done."

Section label:
•	A small header above the completed rows: "Recently Completed" (subtle, grey text).
•	If no orders are completed, this section is hidden entirely.

--------------------------------------------------------------------
DRAG AND DROP — ROW REORDERING
--------------------------------------------------------------------

•	Users can click and drag rows to manually reorder them within the table.
•	A drag handle icon (grip dots or lines) appears on the left side of each row (before the Prio column) to indicate draggability.
•	On drag, the row shows a visual drag preview (slightly elevated, shadow effect).
•	Upon drop, the row is placed in its new position and Prio numbers automatically re-sequence.
•	Upon drop, the scheduling cascade automatically recalculates Start Date/Time and Completion Date for all affected orders within the same feedmill line.
•	Drag and drop only reorders within the current tab/sub-tab view. It does NOT move orders between tabs.
•	Combined orders move together as a group when the lead order is dragged.

Persistence:
•	Reordered positions persist across page refreshes and sessions using browser local storage.
•	Note for future: Implementation should be structured to allow easy migration from local storage to Azure-backed database storage.

NOT available for:
•	Completed orders in the "Recently Completed" section (fixed position)
•	Individual non-lead combined orders (must move with group)

Feature 3: Status-Driven Actions
------------------------------------------------------------------------

All order actions are performed via the Status dropdown column. There are no separate action buttons for Produce, Cancel, Cut, Combine, Mark as Done, Bulk Produce, or Revert. The Actions column contains ONLY the "History" button. 

When a user selects a new status from the dropdown, the system may trigger a modal, validation, or confirmation dialog depending on the selected status. The status change is applied only after the user completes the required flow.

--------------------------------------------------------------------
STATUS-DRIVEN FLOWS
--------------------------------------------------------------------

Status: Any → Cut
Trigger: User selects "Cut" from the dropdown.
Flow:
1.	A Cut modal opens displaying:
-	The order's current details (FPR, Item Description, Suggested Volume).
-	An editable text box: "Volume to keep for this order (MT):"
-	The remaining volume is calculated and displayed in real-time as the user types.
2.	Upon confirmation:
-	The original order retains the specified volume. Its status changes to "Cut." All calculated fields (Number of Batches, Number of Bags, Production Hours) recalculate based on the new volume.
-	A NEW duplicate row is created with the remaining volume. The new row:
•	Copies all original order data (FPR, Material Code, Item Description, FG, SFG, SFG1, Form, etc.)
•	Has the remaining volume (recalculated to be divisible by Batch Size, rounding up).
•	Status = "Cut"
•	FPR Notes auto-populated: "Cut from order FG: [FG value] / SFG: [SFG value] — [MM/DD/YYYY HH:MM AM/PM]"
•	Placed immediately below the original order in the table.
-	The original order's FPR Notes is also updated: "Order cut — [X] MT separated — [MM/DD/YYYY HH:MM AM/PM]"
-	Both orders can independently have their status changed to any other status (In production, Cancel PO, etc.).
-	Cut portions are freely separable (not sticky like combined orders).
-	Scheduling cascade recalculates for all downstream orders.

Status: Any → Combine with other PO
Trigger: User selects "Combine with other PO" from the dropdown.
Flow:
1.	A Combine modal opens displaying:
-	The selected order's details.
-	A list of eligible orders for combining, filtered by the SAME Material Code within the same feedmill line.
-	Each eligible order displays: [Status badge] | Item Description FPR: XXXXXX — XX MT
-	Checkbox selection (light grey border when unselected, green check when selected).
-	There is NO volume limitation or order count limit for combining.
2.	Upon confirmation:
-	All selected orders (including the initiating order) have their status changed to "Combine with other PO."
-	The order with the highest Suggested Volume becomes the LEAD order and receives a "Lead" badge.
-	All combined orders are highlighted with light blue tint row background (#e0f2f8).
-	Combined orders are positioned adjacent to each other (sticky grouping).
-	FPR Notes for each combined order is updated: "Combined with FG: [FG values] / SFG: [SFG values] — MM/DD/YYYY HH:MM AM/PM]"
-	Original order data is preserved on each row (volumes, FPR, etc. remain separate for audit trail).
-	No new combined row is created. The original orders are retained with their original data.
-	For capacity tracking and production calculations, the total combined Suggested Volume is used.
-	Status sync rules apply (see Combined Orders — Row Behavior section in Feature 1).

Status: Any → Cancel PO
Trigger: User selects "Cancel PO" from the dropdown.
Flow:
1.	A confirmation dialog appears:
-	Title: "Cancel Order"
-	Message: "Are you sure you want to cancel this order? Please provide a reason."
-	Text area for cancellation reason (REQUIRED — cannot submit without entering a reason).
-	Placeholder: e.g., "Customer cancelled the order"
-	Buttons: [Confirm Cancel] [Go Back]
2.	Upon confirmation:
-	The order's status changes to "Cancel PO."
-	FPR Notes is appended with: "• Cancelled [MM/DD/YYYY HH:MM AM/PM]: [user's reason]"
-	Order History log records the cancellation.
-	The order remains in the table with "Cancel PO" status (red badge). All fields become read-only except Status.
-	If the order was part of a combine group:
•	Cancelling a non-lead order breaks it out of the group.
•	Cancelling the lead order triggers status sync — all combined orders also cancel (with the same reason).

Status: Cancel PO → Plotted (Reverting a cancellation)
Trigger: User selects "Plotted" from dropdown on a cancelled 
order.
Flow:
1.	A confirmation dialog appears:
-	Title: "Revert Cancelled Order"
-	Message: "Are you sure you want to revert this cancelled order? Please provide a reason."
-	Text area for revert reason (REQUIRED).
-	Placeholder: e.g., "Customer wants the product again"
-	Buttons: [Confirm Revert] [Go Back]
2.	Upon confirmation:
-	Status changes to "Plotted."
-	FPR Notes is appended: "• Reverted [MM/DD/YYYY HH:MM AM/PM]: [user's reason]"
-	All field values are restored to their pre-cancellation state.
-	Fields become editable again per the editability matrix.

Status: Any → In production
Trigger: User selects "In production" from the dropdown.
Flow:
1.	Step 1 — Line Capacity Check:
-	Check if the order's assigned feedmill line has sufficient remaining capacity to accept the order's Suggested Volume (MT).
-	For combined orders, check total combined volume.
-	Capacity is based on orders with Status = "In production" or any "On-going *" status.
-	If insufficient capacity: Show warning: "Cannot change to In Production — [Line X] has insufficient capacity. 
-	Current: [X] MT / 1,000 MT. This order requires [X] MT." 
-	Status reverts to previous value. STOP.
-	If sufficient capacity: Proceed to Step 2.

2.	Step 2 — Minimum MT Threshold Check:
-	Check if the order's Suggested Volume (MT) meets the minimum threshold for its assigned feedmill:
Feedmill 1: 40 MT
Feedmill 2: 20 MT
Feedmill 3: 20 MT
Powermix: 0 MT (exempt — skip to Step 3)
-	For combined orders, use the total combined volume.
-	If BELOW threshold: Show confirmation dialog:
"This order is [X] MT, which is below the recommended minimum of [threshold] MT for [Feedmill Name]. Do you wish to proceed?"
Buttons: [Proceed Anyway] [Cancel]
If "Proceed Anyway": Proceed to Step 3.
If "Cancel": Status reverts to previous value. STOP.
-	If AT or ABOVE threshold: Skip dialog. Proceed to Step 3.

3.	Step 3 — Production Confirmation:
-	Show confirmation: "You are about to move this order to In Production. Confirm?"
-	Buttons: [Confirm] [Cancel]
-	If "Confirm":
•	Status changes to "In production."
•	If part of a combine group and this is the lead order, all combined orders also change to "In production."
•	Line capacity updates.
•	Order History records the action.
-	If "Cancel": Status reverts to previous value. STOP.

Status: In production / On-going * → Done
Trigger: User selects "Done" from the dropdown.
Flow:
1.	The system auto-populates End Date with the current date and time.
2.	Standard confirmation: "You are about to mark this order as done. Confirm?"
Buttons: [Confirm] [Cancel]
3.	Upon confirmation:
-	Status changes to "Done."
-	End Date is set to current date/time (with override mechanism available afterward).
-	The order moves to the "Recently Completed" section at the top of the table.
-	If this makes the completed section exceed 3 orders, the oldest completed order is moved to Order History in Configurations.
-	All fields become read-only except Status and End Date (override).
-	If part of a combine group and this is the lead order, all combined orders also change to "Done."
-	Order History records the action.

Status: Done → In production (Reverting a completion)
Trigger: User selects "In production" from dropdown on a Done order.
Flow:
1.	Override confirmation: "You are about to revert this completed order. Type 'Confirm' to proceed."
Input field: User must type "Confirm" (case-insensitive).
Buttons: [Proceed] (disabled until typed) [Cancel]
2.	Upon confirmation:
-	Status changes to "In production."
-	The order moves from the "Recently Completed" section back to the active orders area.
-	End Date is cleared.
-	Fields become editable per the "In production" editability rules.
-	Order History records the revert.

Status: Any → Hold
Trigger: User selects "Hold" from the dropdown.
Flow:
1.	No modal required. Status changes immediately to "Hold."
2.	Order History records: "Order placed on hold [MM/DD/YYYY HH:MM AM/PM]"
3.	If part of a combine group and this is the lead order, all combined orders also change to "Hold."

Status: Any → On-going batching / On-going pelleting / On-going bagging
Trigger: User selects one of the on-going statuses from the dropdown.
Flow:
1.	No modal required. Status changes immediately.
2.	Order History records the status change.
3.	If part of a combine group and this is the lead order, all combined orders also update.

====================================================================
ORDER HISTORY / AUDIT TRAIL
====================================================================

Every order receives an automatic history entry upon upload. Every significant action performed on an order is automatically logged with a timestamp in an Order History log attached to that order.

ALL orders have history — even if nothing changes after upload, the initial "Uploaded" entry exists.

Actions that generate a history entry:
•	Uploaded — when first imported from SAP file
•	Status changed — ANY status change (e.g., "Plotted → In production", "In production → Done", etc.)
•	Cut — when the order is cut (logged on both portions)
•	Combined — when orders are combined (logged on all orders in the group)
•	Un-combined — when an order breaks out of a combine group
•	Cancelled — includes the user's cancellation reason
•	Reverted from Cancel — includes the user's revert reason
•	Marked as Done — when status changes to "Done"
•	Reverted from Done — when Done order reverts
•	Avail Date overridden — when user overrides the locked date
•	End Date overridden — when user overrides the auto-populated end date
•	Moved to Order History — when archived from the top 3 completed section
•	Hold — when order is placed on hold

History entry format:
•	Timestamp: [MMMM DD, YYYY HH:MM AM/PM]
•	Action: Brief description
•	Details: Additional context

Example:

Order History — FPR: 260302

•	Uploaded [February 24, 2026 08:00 AM]
Imported from SAP file: "planned_orders_feb.xlsx"
Assigned to Feedmill 1 — Line 1
•	Status changed [February 24, 2026 09:15 AM]
Plotted → In production
•	Status changed [February 25, 2026 04:00 PM]
In production → Done
End Date auto-set: February 25, 2026 04:00 PM
•	Moved to Order History [February 26, 2026 08:30 AM]
Archived from Recently Completed section

History Indicator — Colored Dot:
•	Small orange dot (#fd5108), 6-8px, displayed to the left of Item Description text on Line 1.
•	Appears on ALL orders (since all have at least the "Uploaded" entry). The dot indicates history is available for viewing.

History Button (Actions column):
•	Always visible for all orders.
•	Opens a modal with full chronological history log (newest at top).
•	Read-only — users cannot edit or delete history entries.
•	Modal title: "Order History — [FPR]"

•	Tags Item Description — combined with Tags Material Code

--------------------------------------------------------------------
AUTO-POPULATION RULES
--------------------------------------------------------------------

Auto-population occurs upon SAP Order Upload when the Knowledge Base is active. The system matches each order's Material Code (FG) against the Knowledge Base "FG Material Code." If matched:

•	Form → from KB "Form" column
•	Batch Size → from KB based on feedmill assignment:
-	Feedmill 1 → "Batch Size FM1"
-	Feedmill 2 → "Batch Size FM2"
-	Feedmill 3 → "Batch Size FM3"
-	Powermix → "Batch Size PMX"
•	Run Rate → from KB based on specific line number:
-	Line 1 → "Line 1 Run Rate"
-	Line 2 → "Line 2 Run Rate"
-	(and so on for all lines)
•	Material Code (SFG) → from KB "SFG1 Material Code"
•	Threads → from KB "Thread" column
•	Sacks → from KB "[Sacks Material Code] - [Sacks Item Description]"
•	Tags → from KB "[Tags Material Code] - [Tags Item Description]"

All auto-populated fields are non-editable. If KB has blank values for a field, the corresponding order field remains blank.

If no Knowledge Base match is found for an order's Material Code, all KB-dependent fields remain blank and use defaults where applicable (Batch Size defaults to 4).

--------------------------------------------------------------------
KNOWLEDGE BASE MANAGEMENT
--------------------------------------------------------------------

Located in Configurations > Knowledge Base sub-tab.

Upload:
•	Upload button to import Knowledge Base Excel file.
•	Upon upload, the system parses the file and stores it as the active Knowledge Base.
•	The active KB is displayed in a read-only table showing all rows and columns.

Re-apply:
•	A "Re-apply Knowledge Base" button that re-runs the auto-population logic across all existing orders.
•	Use case: When a new or updated Knowledge Base is uploaded, the user can re-apply it to refresh all auto-populated fields.
•	Re-application updates: Form, Batch Size, Run Rate, Material Code (SFG), Threads, Sacks, Tags.
•	Re-application also triggers recalculation of: Suggested Volume, Number of Batches, Number of Bags, Production Hours, Completion Date.

Upload History Log:
•	Displays a chronological log of all Knowledge Base uploads.
•	Each entry shows: file name, upload date/time, number of rows.
•	Read-only reference.

--------------------------------------------------------------------
KNOWLEDGE BASE — CASCADING CALCULATIONS
--------------------------------------------------------------------

When the Knowledge Base is applied or re-applied:

1.	Match Material Code (FG) → KB "FG Material Code"
2.	Populate: Form, Batch Size, Run Rate, Material Code (SFG), Threads, Sacks, Tags
3.	Recalculate Suggested Volume: ceil(Original Volume / new Batch Size) * new Batch Size
4.	Recalculate Number of Batches: Suggested Volume / Batch Size
5.	Recalculate Number of Bags: Suggested Volume / 50 * 1000
6.	Recalculate Production Hours: Suggested Volume / Run Rate
7.	Recalculate Completion Date: Start Date/Time + Production Hours + Changeover Time
8.	Cascade Completion Date to all subsequent orders in the same feedmill line (scheduling cascade)
9.	Recalculate Readiness status for all affected orders
