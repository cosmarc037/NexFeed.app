  ```
## 4. TARGET DATE CALCULATION

For each product in the Next 10 Days file, calculate a "target date" — the date by which the product should be produced to meet inventory without overstocking.

### ALGORITHM

```
function calculateTargetDate(product) {
  const dueForLoading = product.due_for_loading;
  const inventory = product.inventory;
  const dailyValues = product.daily_values; // [{date, value}, ...]
  
  // If due for loading already meets or exceeds inventory, 
  // no production needed
  if (dueForLoading >= inventory) {
    return { targetDate: null, status: 'stock_sufficient' };
  }
  
  // If inventory is 0 or negative, no target
  if (inventory <= 0) {
    return { targetDate: null, status: 'stock_sufficient' };
  }
  
  let cumulative = dueForLoading;
  let previousDate = null;
  
  for (const day of dailyValues) {
    if (day.value === 0 || day.value === null) {
      previousDate = day.date;
      continue;
    }
    
    const newCumulative = cumulative + day.value;
    
    if (newCumulative > inventory) {
      // Would overshoot — target date is the PREVIOUS day
      // (last day before overstocking)
      if (previousDate) {
        return { targetDate: previousDate, status: 'needs_production' };
      } else {
        // First day already overshoots — target date is before first day
        // Use the day before the first date column
        const dayBefore = subtractOneDay(day.date);
        return { targetDate: dayBefore, status: 'needs_production' };
      }
    }
    
    if (newCumulative === inventory) {
      // Exact match — this day is the target
      return { targetDate: day.date, status: 'needs_production' };
    }
    
    cumulative = newCumulative;
    previousDate = day.date;
  }
  
  // Went through all days, never reached inventory
  // Target = last date in the range (produce needed but demand is spread)
  return { 
    targetDate: dailyValues[dailyValues.length - 1].date, 
    status: 'needs_production' 
  };
}
```

**Gallimax 2 Plus Ready Mix:**
```

455.3 + 10.0 (Mar 18) = 465.3
+ 29.0 (Mar 19) = 494.3
+ 51.7 (Mar 20) = 546.0
+ 110.5 (Mar 21) = 656.5
+ 29.0 (Mar 22) = 685.5
+ 10.0 (Mar 23) = 695.5
+ 150.0 (Mar 24) = 845.5 ← EXCEEDS 759.3

Target date = March 23 (last day before overstocking)
```

**Salto Stag Developer:**
```

```
## 5. APPLY TARGET DATES TO SAP ORDERS

When both SAP orders and the Next 10 Days file are uploaded, the app matches them and applies inferred target dates to non-dated orders.

### MATCHING RULES

**Match by:** Material Code (material_code_fg in SAP = Material Code in Next 10 Days)

**Apply ONLY when:**
- The order's avail date is a NON-DATE value: "prio replenish", "safety stocks", "for sched", or any other non-date text.
- If the order has an ACTUAL avail date (e.g., "March 22, 2026"), IGNORE the Next 10 Days data — use the existing avail date for sequencing.

**What happens when matched:**
- The order receives an INFERRED target date internally.
- The displayed avail date in the table REMAINS as "prio replenish" or "safety stocks" — the original value is NOT overwritten.
- The inferred target date is shown as a SECONDARY reference line below the avail date.
```

```
## 6. AUTO-SEQUENCE — USE TARGET DATES FOR PRIORITIZATION

Update auto-sequence and auto-sort logic to use inferred target dates for non-dated orders.

### PRIORITY HIERARCHY FOR SEQUENCING

```
1. Orders with actual avail dates (from SAP) — sorted chronologically
2. Orders with inferred target dates (from Next 10 Days) — sorted 
   chronologically, INTERLEAVED with actual dated orders based on date
3. Non-dated orders WITHOUT an inferred target date — placed after 
   all dated/targeted orders
4. Orders with "Stock sufficient" flag — placed at the very bottom
```

### INTERLEAVING EXAMPLE

```
Orders:
- Order 1: Avail = March 22 (actual)
- Order 2: Avail = prio replenish, Target = March 19 (inferred)
- Order 3: Avail = safety stocks, Target = March 25 (inferred)
- Order 4: Avail = March 23 (actual)
- Order 5: Avail = March 24 (actual)
- Order 6: Avail = prio replenish, No target (no match)
- Order 7: Avail = safety stocks, Stock sufficient

Sorted sequence:
Prio 1: Order 2 — prio replenish (Target: Mar 19) ← earliest
Prio 2: Order 1 — March 22 (actual)
Prio 3: Order 4 — March 23 (actual)
Prio 4: Order 5 — March 24 (actual)
Prio 5: Order 3 — safety stocks (Target: Mar 25)
Prio 6: Order 6 — prio replenish (no target, no priority)
Prio 7: Order 7 — safety stocks (stock sufficient, lowest)
```

### DRAG-AND-DROP ENFORCEMENT

Orders with inferred target dates follow the SAME chronological drag-and-drop rules as actual dated orders:

- Target: March 19 CANNOT be placed after Avail/Target: March 22.
- Target: March 25 CANNOT be placed before Avail/Target: March 23.
- Both actual avail dates and inferred target dates are considered together in the chronological sequence.
- Non-dated orders WITHOUT target dates remain freely movable.
```
