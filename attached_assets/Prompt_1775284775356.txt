## FIX 1: EDITED CELLS STAY IN THEIR ORIGINAL ROW POSITION

When editing a cell and saving, the row should stay at its original position in the table. Do not re-sort or re-order rows after editing.

### THE PROBLEM

After editing a cell, the table re-renders and the edited row moves to a different position — likely because the table is being re-sorted by a key like FG Material Code or some other field after state update.

### THE FIX

Ensure the table does NOT re-sort after cell edits. Maintain the original row order.

```javascript
function handleCellChange(rowId, field, newValue) {
  setKbData(prev => {
    // Map over existing array — preserves order
    return prev.map(row => {
      if (row.id === rowId) {
        const oldValue = row[field];
        if (oldValue !== newValue) {
          trackChange('modify', row, field, oldValue, newValue);
        }
        return { ...row, [field]: newValue };
      }
      return row;
    });
  });
}
```

### THE FIX

  // Assign stable IDs
  const dataWithIds = assignStableIds(parsedData);
  
  // Set as current KB data
  setKbData(dataWithIds);
  
  // Create history entry with FULL snapshot of the uploaded data

    id: Date.now(),
    type: 'file_upload',
    timestamp: new Date().toISOString(),
    summary: `Uploaded new file: ${filename} (${recordCount} records)`,

    snapshot: JSON.parse(JSON.stringify(dataWithIds)),  // DEEP COPY of uploaded data

  addToHistory(uploadEntry);
