## OBJECTIVE

We need to enforce a production-aware order flow in the table.

Current status options include:
- Plotted
- Planned
- Hold
- Cut
- In production
- On-going batching
- On-going pelleting
- On-going bagging
- Done
- Cancel PO

We want to improve this so that:
1. order placement follows a strict operational flow
2. on-going production orders are treated as actively being produced
3. not-yet-started orders cannot appear above currently-produced orders
4. "In production" is no longer used as a standalone row status
5. the Overview tab can still count all On-going* orders as "In Production"

## 1. DEFINE THE STATUS FLOW / TABLE BLOCK ORDER

The order table should follow this top-to-bottom placement hierarchy:

1. Done
2. On-going bagging
3. On-going pelleting
4. On-going batching
5. Plotted / Planned / Hold / Cut / Combined with other PO
6. Cancel PO

This means:
- completed orders remain above all others
- currently running orders come next
- not-yet-started orders come after currently running orders
- cancelled orders remain at the bottom

Remove "In production" as a selectable/displayed row status.

Keep only the specific statuses:
- On-going batching
- On-going pelleting
- On-going bagging

These should be the statuses used in the order rows.

function isInProductionStatus(status) {
  const normalized = String(status || '').trim().toLowerCase();
  return [

    'on-going pelleting',
    'on-going bagging',

}

## 4. ENFORCE FIXED STATUS GROUP PLACEMENT IN THE TABLE

Create a status-priority ranking function:

```javascript
function getStatusFlowRank(status) {
  const s = String(status || '').trim().toLowerCase();

  if (s === 'done') return 1;
  if (s === 'on-going bagging') return 2;
  if (s === 'on-going pelleting') return 3;
  if (s === 'on-going batching') return 4;
  if (['plotted', 'planned', 'hold', 'cut', 'combined with other po'].includes(s)) return 5;
  if (s === 'cancel po') return 6;

  return 5;
}
```

Use this ranking to ensure the table never renders:

## 6. VALIDATE ON-GOING STATUS CHANGES AGAINST PRIOR ORDERS

Users should not be able to change an order to:
- On-going batching
- On-going pelleting
- On-going bagging

if a prior order above it is still in a not-yet-started state, such as:
- Plotted
- Planned
- Hold
- Cut

Because this would imply the user is starting a later order while an earlier one is still waiting.

### Rule:
When user selects an On-going* status:
- check orders above the target order
- if any earlier order is still not yet in production and not done/cancelled, show warning and block the change

### Suggested helper:
```javascript
function isNotYetStartedStatus(status) {
  const s = String(status || '').trim().toLowerCase();
  return ['plotted', 'planned', 'hold', 'cut', 'combined with other po'].includes(s);
}

function canSetOnGoingStatus(orders, targetIndex) {
  const priorOrders = orders.slice(0, targetIndex);

  return !priorOrders.some(order => isNotYetStartedStatus(order.status));
}
```
