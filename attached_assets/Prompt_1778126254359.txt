Revisit and fix the row-level sequence insight for the Standard Sequence so it updates when “Apply Profitability (Margin)” changes the order positions.

---

## OBJECTIVE

We added an “Apply Profitability (Margin)” action to the order preview.

This feature re-sorts the Standard Sequence by prioritizing higher-margin orders while still respecting order dates and sequencing constraints.

The reordering itself is working, but the row-level sequence insights are not updating accordingly.

Currently:
- the order positions change after profitability is applied
- but the insights still explain the old pre-profitability sequence logic
- so the insight text no longer matches the actual position of the order in the final Standard Sequence

We need to make sure the row-level insight is aligned with what the Standard Sequence actually did after profitability was applied.

---

## PROBLEM TO FIX

After profitability is applied:
- order A may move above order B because it has a higher margin
- but the insight still says something like:
  - protected by avail date
  - grouped by color/category
  - due to MTO protection
  - original ordinal explanation like Fourth/Fifth/Sixth

This causes a mismatch because:
- the final position changed
- but the explanation did not

The insight should always reflect the final sequence state currently being shown.

---

## REQUIRED BEHAVIOR

When “Apply Profitability (Margin)” changes the Standard Sequence:
- regenerate the row-level insights for the Standard Sequence
- use the final profitability-adjusted order positions
- explain why that order is now in that position

The new explanation should still respect existing rules like:
- availability date protection
- Future Dispatches/date constraints
- MTO constraints if applicable

But if margin was the deciding factor within those allowed rules, the insight should say so.

---

## 1. TREAT "STANDARD SEQUENCE + PROFITABILITY APPLIED" AS A DISTINCT FINAL STATE

Right now, the app appears to update only the row order, but not the reasoning.

After profitability is applied, the Standard Sequence should be treated as a new final state:
- same strategy card = Standard Sequence
- but with profitability-adjusted ranking

That means:
- rows
- ordinal positions
- sequence insights
must all be recomputed from the new final order arrangement

Do not keep old row-level explanations from the pre-profitability state.

---

## 2. REGENERATE ROW-LEVEL INSIGHTS AFTER PROFITABILITY REORDERING

After the profitability sort is applied:
- rebuild the row-level sequence explanations for the new order list
- update ordinal wording like:
  - First
  - Second
  - Third
  - etc.
- update the reasoning based on the new actual neighboring context and final placement

### Required flow
```javascript
const profitabilityAdjustedOrders = applyProfitabilityOrdering(standardSequenceOrders);

const updatedInsights = generateStandardSequenceInsights({
  orders: profitabilityAdjustedOrders,
  profitabilityApplied: true,
});
```

The UI should then render these new insights, not the stale ones.

---

## 3. INCLUDE PROFITABILITY AS A REASON ONLY WHEN IT ACTUALLY AFFECTED POSITION

Do not blindly add “higher margin prioritized” to every row.

Instead:
- detect whether profitability changed the order’s relative placement
- if yes, mention profitability/margin as part of the explanation
- if no, keep the existing standard-sequence explanation

### Example:
If two orders had the same date constraints and were both eligible to swap, and the higher-margin one was moved ahead:
- mention profitability as a deciding factor

If an order stayed in place because date constraints were stricter than margin:
- keep the explanation date-driven

---

## 4. PRESERVE DATE / MTO / FUTURE DISPATCHES PROTECTION IN THE EXPLANATION

Profitability should not erase important scheduling constraints from the explanation.

The insight should still reflect:
- avail date protection
- Future Dispatches protection
- MTO constraints
- sequence safety
- grouping/changeover factors if still relevant

But if profitability caused the final ordering among otherwise valid candidates, include it clearly.

### Good example:
```text
Fifth, protected by avail date 2026-05-23, then prioritized ahead of lower-margin orders within the same allowable window.
```

### Another example:
```text
Seventh, kept within the 2026-05-23 availability boundary and promoted within that group due to higher margin.
```

This is much more aligned to the actual post-profitability ordering.

If profitability changes the order positions:
- recompute the ordinal label
- do not keep the old ordinal from the previous sequence state
