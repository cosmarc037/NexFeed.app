Please update the Order History feature so it remains simple and focused, while adding support for the most meaningful movement/edit events introduced in the app. Also update the new history entries to use a two-line format: a clear main action line plus a sub-text/details line where applicable.

---

## OBJECTIVE

Enhance the History column / OrderHistoryModal so it records only a small set of clear, high-signal operational events.

We want to keep history:
- simple
- readable
- easy to scan

Do not flood the history modal with planning noise or verbose strategy explanations.

---

## KEEP EXISTING HISTORY BEHAVIOR

Keep the current history system and data structure as-is:

```json
{
  "timestamp": "05/09/2026, 02:15 PM",
  "action": "Status changed: plotted → completed",
  "details": "End Date auto-set: 05/09/2026, 02:15 PM"
}
```

Continue using:
- `action` = main visible line
- `details` = sub-text / smaller grey text underneath

The modal should continue to:
- display newest first
- show timestamp
- show action
- show details underneath when present

Do not remove or break existing history events such as:
- Uploaded
- Status changed
- Cancel / restore
- Completed
- Cut order / merge back
- Un-combine
- etc.

This is an additive enhancement only.

---

## NEW HISTORY EVENTS TO SUPPORT

Only add these new history event types:

1. **Auto-sequence**
2. **Line shutdown**
3. **Feedmill shutdown**
4. **Override volume**
5. **Manually reordered**
6. **Optimize order**

Do not add:
- apply to schedule / strategy application
- profitability applied
- auto-sequence reasoning summary
- app-adjusted divisible volume
- generic planning rationale
- sequence insight explanation

These are intentionally excluded to keep history clean and useful.

## DISPLAY FORMAT

Use a two-line format for the new movement/shutdown entries:

### Main action line
Shows the primary change

### Sub-text/details line
Shows supporting context such as:
- AI strategy name
- shutdown reason
- priority movement

This should visually fit the current modal style shown in the screenshot:
- bold action
- small grey details underneath

## 1. AUTO-SEQUENCE HISTORY

When auto-sequence changes an order and that change is actually committed/applied, add a history entry.

### If the order moved across lines:
**Action**
- `Auto-sequence: Line X -> Line Y`

**Details**
- `[AI-powered sequence name]. Moved from Prio A -> B`

### Example
Action:
- `Auto-sequence: Line 1 -> Line 3`

Details:
- `Red-Last Changeover Relief. Moved from Prio 5 -> 3`

### If the order stayed on the same line but changed priority:
**Action**
- `Auto-sequence: Prio A -> B`

**Details**
- `[AI-powered sequence name]`

### Example
Action:
- `Auto-sequence: Prio 7 -> 3`

Details:
- `Swine Diameter Segmentation`

Use the actual AI strategy name selected/applied by the user.

---

## 2. LINE SHUTDOWN HISTORY

When a line shutdown diverts an order to another line, add a history entry.

A line shutdown always implies cross-line movement.

### Action
- `Line shutdown: Line X -> Line Y`

### Details
- `[selected shutdown reason]. Moved from Prio A -> B`

### Example
Action:
- `Line shutdown: Line 2 -> Line 3`

Details:
- `Scheduled maintenance. Moved from Prio 3 -> 5`

### Important
The shutdown reason must come from the actual shutdown modal selection (the reason dropdown), such as:
- Scheduled maintenance
- Equipment breakdown
- Natural disaster
- Power outage
- Supply shortage
- Staff shortage
- Quality issue
- Regulatory compliance
- Other

Use the exact selected reason text in history.

For now, do not include optional shutdown notes in the history entry unless separately requested later.

---

## 3. FEEDMILL SHUTDOWN HISTORY

When a feedmill shutdown diverts an order to another line, add a history entry.

### Action
- `Feedmill shutdown: Line X -> Line Y`

### Details
- `[selected shutdown reason]. Moved from Prio A -> B`

### Example
Action:
- `Feedmill shutdown: Line 4 -> Line 7`

Details:
- `Power outage. Moved from Prio 2 -> 6`

As with line shutdown, use the actual selected shutdown reason from the shutdown modal.

Also include the priority movement in the details.

---

## 4. OVERRIDE VOLUME HISTORY

When a user manually overrides the volume, add a history entry.

### Action
- `Override volume: OLD MT -> NEW MT`

### Example
- `Override volume: 140 MT -> 160 MT`

### Details
- leave blank / not required for now

### Important
This applies only to **manual override volume**.

Do **not** add app-adjusted divisible volume into history.

App-adjusted divisible volume should remain outside the history modal.

---

## 5. MANUALLY REORDERED HISTORY

When a user manually drags/drops an order within the same line, add a history entry.

This is always same-line only.

### Action
- `Manually reordered: Prio A -> B`

### Example
- `Manually reordered: Prio 7 -> 3`

### Details
- leave blank / not required for now

Do not include line movement in this event.

---

## 6. OPTIMIZE ORDER HISTORY

When the user uses the **Optimize** feature and the app moves an order from another line into the selected line, add a history entry.

### Action
- `Optimize order: Line X -> Line Y`

### Details
- `Moved from Prio A -> B`

### Example
Action:
- `Optimize order: Line 3 -> Line 4`

Details:
- `Moved from Prio 4 -> 3`

This should be distinct from Auto-sequence because Optimize is a separate user-triggered feature.

---

## 7. APP-ADJUSTED DIVISIBLE VOLUME SHOULD USE A TOOLTIP, NOT HISTORY

If a volume is app-adjusted because the original value is not divisible by batch size:
- do not write it to order history
- instead, surface it via a tooltip in the order table if needed

Example tooltip:
- `Adjusted from 145 MT to 148 MT so the volume is divisible by batch size 4.`

This is intentionally outside history.

---

## 8. WRITE HISTORY ONLY WHEN THE CHANGE IS ACTUALLY COMMITTED

For all new history types:
- write the history entry when the action is actually committed
- not when it is merely previewed

Examples:
- Auto-sequence entry should be written when the selected sequence is applied
- Optimize entry should be written when optimize is executed
- Shutdown entries should be written when the divert/reassignment is confirmed
- Manual reorder should be written when the drag/drop reorder is saved/applied
- Override volume should be written when the manual override is confirmed

This keeps history as a true audit trail.
