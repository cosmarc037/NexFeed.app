Please update the **Start Date / Start Time resolution logic for downstream orders** with the following validation and auto-fill behavior.

## 1. APPLY THIS CHANGE ONLY TO 2ND ORDERS ONWARD

Please apply this logic only to:
- the **2nd order onward**
- or any order that already has a **previous order** above it in the sequence

Do **not** change the already-implemented fallback/default behavior for the **first order**.

### Expected result
The first-order logic remains as-is.

This update only affects orders whose scheduling logic can reference a **previous order’s estimated completion datetime**.

## 2. IF USER SETS A START DATE BUT NO START TIME, DEFAULT THE TIME TO 8:00 AM

If a user manually sets a **Start Date** for an order, but leaves **Start Time** blank:

- keep the user-selected **Start Date**
- automatically set **Start Time = 8:00 AM**

### Expected result

- `Start Date = selected date`
- `Start Time = 8:00 AM`

## 3. VALIDATE START DATE SELECTION FOR DOWNSTREAM ORDERS

When a user manually sets a **Start Date** for any order that has a previous order, please enforce the following date restrictions.

### A. Do not allow past dates
The user must not be able to select any date earlier than the current date.

For example, if today is:
- `May 23, 2026`

the user must not be able to select:
- `May 19, 2026`
- `May 20, 2026`
- `May 21, 2026`
- `May 22, 2026`

### B. Do not allow the same date as the previous order’s completion date
If the previous order’s estimated completion date is:
- `May 23, 2026`

then the user should **not** be allowed to manually select:
- `May 23, 2026`

for the next order’s Start Date.

### Reason
That same-day scheduling case should already be resolved by sequence logic and/or the Start Time rule.

If the user is explicitly setting a **Start Date** for a downstream order, it should represent a valid future scheduling date, not:
- a past date
- or the same date already occupied by the previous order’s completion date

### Expected result
For downstream orders, allowed Start Date choices must be:
- not in the past
- and strictly later than the previous order’s completion date’s calendar date

---

## 4. IF USER SETS A START TIME BUT NO START DATE, DERIVE THE DATE FROM THE PREVIOUS ORDER’S ESTIMATED COMPLETION

If a user manually sets a **Start Time** for an order, but leaves **Start Date** blank:

- look at the **previous order’s estimated completion datetime**
- use that as the reference point for deriving the Start Date

### Resolution logic
Compare the user-entered Start Time against the **time-of-day** of the previous order’s estimated completion.

#### If the entered Start Time is later than or equal to the previous order’s completion time
Use the **same date** as the previous order’s estimated completion date.

#### If the entered Start Time is earlier than the previous order’s completion time
Use the **next calendar date** after the previous order’s estimated completion date.

---

- `Start Date = May 23, 2026`

- `Start Date = blank`

Since `10:00 AM` is earlier than `12:00 PM`, the system should resolve to:
- `Start Date = May 24, 2026`

### Expected result
A manually entered Start Time should still be respected, but the Start Date should be intelligently inferred based on sequence continuity.
