Please update the **auto-sequence preview handling for Start Date / Start Time** so that system-carried timing values are reset after resequencing, while manually set timing values are preserved and flagged if they no longer fit the new schedule.

---

## 1. RESET NON-MANUAL START DATE / TIME VALUES IN AUTO-SEQUENCE PREVIEW

The application currently shows Start Date / Start Time values in the order table, and the first order can receive a default start basis when no manual timing is set.

However, after **auto-sequence**, some Start Date / Start Time values appear to be carried over into the preview in a way that can be misleading or contradictory to the newly generated sequence.

Please update this behavior.

### New rule
After auto-sequence is generated, the preview should **not blindly carry over system-derived / inherited Start Date and Start Time values** for resequenced orders.

### Expected result
In the auto-sequence preview:
- only the **first order** should retain/display the start basis generated by the app’s default rule
- later orders should not keep old carried-over Start Date / Start Time values unless those values were manually set by the user

---

## 2. KEEP THE FIRST ORDER’S START DATE / TIME AS THE BASIS

Please preserve the existing behavior where the first order can have a Start Date / Start Time basis generated by the app according to its default rules.

### Goal
The auto-sequence preview should still have a clear starting anchor for the schedule.

### Expected result
After auto-sequence:
- the first order continues to show its Start Date / Start Time basis
- this acts as the schedule anchor for the preview

---

## 3. PRESERVE MANUALLY SET START DATE / TIME VALUES

If a user has manually entered a:
- Start Date
- Start Time
- or both

for an order, please preserve those manual values in the auto-sequence preview even after resequencing.

### Important
Manual user input should not be erased just because the order was moved or resequenced.

### Expected result
Manually set Start Date / Start Time values should still appear in the preview after auto-sequence.

---

## 4. DIFFERENTIATE BETWEEN SYSTEM-DERIVED VALUES AND MANUALLY SET VALUES

Please clearly distinguish between:
- Start Date / Start Time values that were manually set by the user
- Start Date / Start Time values that were only system-derived, inherited, or displayed because of existing sequence/default logic

### Goal
The preview reset logic should only clear/reset:
- non-manual / system-carried values

It should preserve:
- true manual values

### Expected result
The app should know which timing values were explicitly set by the user and which were merely the result of prior sequence calculation/display logic.

---

## 5. SHOW A WARNING ICON IF A PRESERVED MANUAL START DATE / TIME NO LONGER FITS THE NEW SEQUENCE

If a manually set Start Date / Start Time is preserved in the auto-sequence preview, but that timing no longer makes sense in the newly resequenced order flow, please show a warning icon.

### Goal
Let the user keep their manual input visible, but alert them that it may now be misaligned with:
- the new sequence
- avail dates
- current schedule flow
- upstream/downstream timing
- expected completion logic

### Expected result
Manually set values remain visible, but a warning icon appears if the value is no longer applicable to the current sequence context.

---

## 6. USE THE WARNING TO SIGNAL THAT THE USER MAY NEED TO MANUALLY ADJUST THE TIMING

Please use the warning icon as a signal that the manual timing should be reviewed.

### Suggested tooltip/message
- `Manual start date/time may no longer align with the current sequence.`
- `This manually set start date/time may conflict with the updated schedule.`
- `Start date/time was manually set and should be reviewed after auto-sequence.`

### Expected result
Users understand that:
- the system preserved their manual input intentionally
- but they may need to edit it because the sequence has changed

---

## 7. DO NOT DISPLAY STALE CARRIED-OVER START DATE / TIME VALUES FOR NON-MANUAL ORDERS

Please make sure that orders without manually set Start Date / Start Time do not continue showing stale carried-over timing values from the old sequence after auto-sequence is run.

### Goal
Avoid creating the impression that those times are still explicitly valid after resequencing.

### Expected result
For non-manual rows in the preview:
- Start Date / Start Time should reset/clear
- unless the row is the first order and is using the schedule basis

---

## 8. APPLY THIS TO THE AUTO-SEQUENCE PREVIEW / AFTER TABLE

Please apply this logic specifically to the:
- auto-sequence preview
- after-table / resequenced result view

### Expected result
The preview should reflect:
- one clear starting basis
- preserved manual timing where applicable
- warnings for preserved manual timing that no longer fits
- no misleading carried-over timing for other rows

## 9. PRESERVE EXISTING SCHEDULING LOGIC OUTSIDE THIS PREVIEW BEHAVIOR

Please preserve the existing scheduling logic for:

- downstream estimated completion calculations
- schedule warnings already in place
- other start-date/start-time validation rules

### Expected result
