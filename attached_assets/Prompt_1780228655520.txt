# 2. ADD A NEW COLUMN: `VOLUME TO PRODUCE (MT)`

Please add a new column in the **Re-Order Details** section called:

- `Volume to Produce (MT)`

### Goal
This column should represent the actual production quantity for the generated re-order, which is different from the original source order volume.

### Expected result
Users can clearly see the distinction between:
- the original/source order volume
- and the generated replenishment volume to be produced

---

# 3. THE RE-ORDER SHOULD NOT PRODUCE THE DEDUCTED SOURCE ORDER VOLUME

Please update the fulfillment logic so that when a user approves a suggested re-order, the system does **not** generate the new order using the original source order’s volume.

### Important rule
The generated re-order should **not** simply reproduce the same volume as the source order that was fulfilled from inventory.

### Reason
If the feature simply regenerates the exact original volume at a later date, it defeats the purpose of using inventory to fulfill the original demand quickly.

### Expected result
The re-order suggestion should behave as a replenishment order, not as an exact duplicate of the original fulfilled order.

---

# 4. USE `AVERAGE DAILY` AS THE BASIS FOR `VOLUME TO PRODUCE`

Please use the product’s **Average Daily** value as the basis for the generated re-order’s **Volume to Produce (MT)**.

### Example
If:
- Source order volume = `80 MT`
- Inventory in Demo Future Dispatches = `90 MT`
- Average Daily = `20 MT`

and the user approves the suggestion,

then:
- the source order is treated as fulfilled using inventory
- the source order’s `80 MT` is deducted from the inventory logic
- but the generated re-order’s `Volume to Produce` should be:
  - `20 MT`
  - because it is based on **Average Daily**
  - not the original `80 MT`

### Expected result
The generated re-order quantity reflects replenishment need based on Average Daily, not the original fulfilled volume.

---

# 5. WHEN THE USER APPROVES, DEDUCT THE SOURCE ORDER VOLUME FROM INVENTORY / ADJUSTED INVENTORY

Please preserve and align the inventory logic such that when the user approves the suggested re-order:

- the original/source order is treated as fulfilled using inventory
- the source order volume is deducted from the product’s inventory state
- the `Adjusted Inventory` reflects that deduction

### Important
This deduction should still use the **source order volume**, because that is the amount assumed to be fulfilled using stock.

### Expected result
The inventory side reflects the original order fulfillment, while the generated re-order quantity remains based on Average Daily.

---

# 6. KEEP THE SOURCE ORDER VOLUME AND RE-ORDER VOLUME AS SEPARATE CONCEPTS

Please clearly preserve the difference between:

### Source Order Volume
- the original order quantity in the schedule
- used when determining whether inventory can satisfy that order
- deducted from inventory when approval happens

### Volume to Produce
- the generated replenishment/re-order quantity
- based on **Average Daily**
- used to create the actual new re-order in the sequence

### Goal
Avoid confusion between the demand being fulfilled from stock and the replenishment quantity being generated.

### Expected result
These two values should remain separate in both logic and UI.

---

# 7. USE `VOLUME TO PRODUCE` AS THE BASIS FOR PRODUCTION TIME CALCULATION

Please update the production-time calculation for the generated re-order so that it uses:

- `Volume to Produce (MT)`

and **not** the original source order volume.

### Example
If:
- source order = `80 MT`
- average daily / volume to produce = `20 MT`

then production time should be calculated using:
- `20 MT`

not:
- `80 MT`

### Expected result
The displayed `Production Time` for the re-order suggestion correctly reflects the actual generated replenishment quantity.

---
