Please refine the **AI Avail Date** and **approval flow** for **Suggested Re-orders** in the **Fulfillment (Demo)** workspace so that the AI date is **actually AI-generated**, and approval includes a placement/impact confirmation step before insertion into the schedule.

---

# 1. MAKE `AI AVAIL DATE` ACTUALLY AI-GENERATED

The current `AI Avail Date` is not truly AI-generated.

It is currently derived from the source order’s existing:
- `target_avail_date`
- or fallback `original_avail_date`

This is misleading if the field is labeled as AI-driven.

### New requirement
Please update the logic so that the `AI Avail Date` is actually generated by AI / sequence reasoning, not simply copied from the source order’s existing availability date.

### Expected result
The displayed `AI Avail Date` should reflect an AI-evaluated recommendation, not just a reused field from the source order.

---

# 2. USE AI / SEQUENCE REASONING TO DETERMINE THE BEST RE-ORDER DATE AND POSITION

Please make the AI evaluate the suggested re-order using real sequence context.

### The AI should consider:
- the current lineup of orders on the target line
- current order sequence / queue state
- changeovers
- the best insertion position for the re-order
- whether placing the re-order in that position may delay downstream orders
- whether the insertion is operationally reasonable
- sequence impact / feasibility

### Goal
The system should recommend a realistic production target/placement for the re-order based on the actual current schedule.

### Expected result
The suggested date and placement should be informed by real queue/sequence context, not simply inherited from the source order.

---

# 3. TREAT THE AI AVAIL DATE AS A TRUE RECOMMENDED PRODUCTION TARGET

Please treat the generated `AI Avail Date` as the AI’s recommended timing target for producing the re-order.

### Goal
This date should help answer:
- when should this replenishment order ideally be produced?
- where should it be inserted so it fits the current schedule without creating unnecessary downstream disruption?

### Expected result
The AI Avail Date becomes a meaningful planning recommendation tied to insertion strategy.

---

# 4. DO NOT JUST REUSE THE SOURCE ORDER’S TARGET / ORIGINAL AVAIL DATE AS THE DISPLAYED AI DATE

Please remove the current logic that directly repurposes:

- `target_avail_date`
- `original_avail_date`

as the displayed AI Avail Date.

### Important
Those fields may still be useful as inputs/context for the AI, but they should not be directly reused as the final AI date without actual AI sequence analysis.

### Expected result
The final displayed AI Avail Date should be a true generated recommendation, not a copied source-order field.

---

# 5. WHEN THE USER CLICKS `APPROVE`, SHOW A CONFIRMATION MODAL BEFORE INSERTION

Please add another confirmation step when the user approves a suggested re-order.

### New flow
When the user clicks:
- `Approve`

do **not** insert the re-order immediately.

Instead:
- open a confirmation dialog/modal first
- show the proposed insertion recommendation and impact analysis
- then let the user confirm the insertion

### Expected result
The user can review the proposed re-order placement before it is committed into the schedule.

---

# 6. SHOW KEY RE-ORDER DETAILS IN THE APPROVAL CONFIRMATION MODAL

The confirmation dialog/modal should display the key details of the suggested re-order, such as:
- item description
- FPR
- material codes
- original/source order volume
- re-order volume to produce
- batch size
- production time
- inventory context if useful
- AI Avail Date
- target line

### Goal
Users should have enough information to understand what they are about to insert into the schedule.

### Expected result
The approval modal gives a concise but informative summary of the generated re-order.

---

# 7. SHOW THE PROPOSED INSERTION POSITION IN THE APPROVAL MODAL

Please include the suggested insertion position in the modal.

### Examples
Show:
- target line
- proposed priority / insertion position
- where in the current queue the re-order would be placed

### Goal
Users should know where the re-order is expected to land in the schedule.

### Expected result
The modal clearly explains the intended insertion position before the user confirms.

---

# 8. SHOW AN IMPACT ANALYSIS IN THE APPROVAL MODAL

Please add a simple impact analysis in the approval confirmation modal, similar in spirit to other schedule-impact review dialogs in the app.

### The impact analysis should include relevant factors such as:
- whether downstream orders will shift
- whether changeover impact is introduced or increased
- whether the insertion is expected to cause delay to downstream orders
- whether the insertion is considered safe/reasonable
- any scheduling tradeoff that is important for user review

### Goal
The user should understand the operational impact of approving the re-order placement.

### Expected result
The modal gives the user confidence that the AI is recommending a reasonable placement.

---

# 9. ENSURE THE AI RECOMMENDATION AVOIDS CAUSING UNNECESSARY DOWNSTREAM DELAYS

When generating the AI Avail Date / insertion recommendation, please bias the recommendation toward positions that do **not** cause unnecessary downstream delays.

### Important
The AI does not need to promise zero impact in every case, but it should try to:
- minimize disruption
- avoid avoidable downstream delay
- avoid poor placements that make the sequence worse

### Expected result
The AI recommendation should be operationally responsible and not simply choose any arbitrary fitting slot.

---

# 10. AFTER USER CONFIRMS, INSERT THE RE-ORDER USING THE REVIEWED AI RECOMMENDATION

Once the user reviews the modal and confirms, then insert the re-order into the schedule using the AI-reviewed placement result.

### Expected result
The final inserted position should match the recommendation shown in the approval modal.

### Goal
Avoid cases where:
- the modal says one insertion point
- but the actual applied result lands somewhere else

---

# 11. APPLY THIS TO THE FULFILLMENT (DEMO) WORKFLOW ONLY

Please apply this change specifically to the **Suggested Re-orders** approval flow in the **Fulfillment (Demo)** workspace.

### Expected result
This becomes part of the demo replenishment workflow without changing the main application behavior unless explicitly requested later.
