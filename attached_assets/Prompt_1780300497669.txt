Please refine the **auto-sequence preview production time display and calculation** so that the preview shows the actual run rate used, and uses that same run rate as the basis for production-time calculation.

---

## 1. ADD `RATE` UNDER THE PRODUCTION TIME IN THE PREVIEW

Please update the **Production Time** column in the auto-sequence preview so that the supporting details below the time include both:

- `CO: xx`
- `Rate: xx`

### Important formatting requirement
Please show them on separate lines under the Production Time value.

### Required display format
Use this structure:

- `[Production Time value]`
- `CO: xx`
- `Rate: xx`

### Example
- `7.75 hours`
- `CO: 0.33`
- `Rate: 19.10`

### Do NOT format it like:
- `CO: 0.33 | Rate: 19.10`

### Expected result
The preview should clearly show both the changeover and the run rate used for that row, each on its own line.

---

## 2. USE THE DISPLAYED `RATE` AS THE BASIS FOR PREVIEW PRODUCTION TIME

Please make sure the **Rate** shown under the preview Production Time is the exact same run rate used to calculate the preview production time.

### Required rule
Preview production time must be based on:

- `Volume / displayed Rate`

### Expected result
There should be no case where the preview shows one run rate but uses a different hidden/default feedmill rate to calculate production time.

---

## 3. DO NOT FALL BACK TO GENERIC FEEDMILL RUN-RATE VALUES IF A ROW-SPECIFIC RATE EXISTS

There are cases where the app appears to calculate preview production time using generic/default feedmill rates such as:
- `20 MT/hr` for FM1
- `10 MT/hr` for FM2 / FM3 / PMX

even when a row-specific rate should be used.

Please prevent that behavior in the preview.

### Expected result
If the order/row has an actual applicable run rate, the preview should:
- display that rate
- and use that same rate for production-time calculation

### Goal
Avoid production-time mismatches caused by hidden/default feedmill fallback rates.

---

## 4. MAKE THE PREVIEW MORE TRANSPARENT BY SHOWING THE RATE USED

The reason for adding `Rate` under Production Time is to make the preview more transparent and easier to validate.

### Goal
Users should be able to visually confirm that:
- the shown production time
- makes sense for the shown volume
- using the shown rate

### Expected result
The preview becomes easier to trust because the basis for the production time is visible.

---

## 5. KEEP THE FINAL SCHEDULE’S PRODUCTION-TIME LOGIC CONSISTENT

If the final schedule already correctly shows:
- production time
- CO
- Rate

please preserve that correct behavior.

### Goal
Align the preview to the same row-specific logic and transparency already expected in the schedule view.

### Expected result
Preview and schedule should feel consistent in both:
- calculation basis
- supporting displayed values

---

Please ensure this improvement applies across:
