## 1. USE THE LEAD ORDER AS THE CANONICAL BASIS FOR COMBINED-ORDER SEQUENCING

There appears to be an issue where combined orders may be sequenced using the avail dates or metadata of sub-orders, which can distort the final placement.

### Required rule
For a combined order / combined production entity, please use the **lead order** as the canonical basis for sequencing.

### This should apply especially to:
- avail date logic
- AI rank context
- final rank / sequencing placement
- date-based ordering behavior
- any combined-entity metadata used during preview/apply sequencing

### Expected result
Combined orders should behave like one unified production entity, with the lead order acting as the single canonical reference.

---

## 2. DO NOT LET SUB-ORDER AVAIL DATES OVERRIDE THE COMBINED ENTITY’S SEQUENCING BASIS

The current behavior suggests the app may be using sub-order avail dates when deciding where a combined order should land.

### Current problem
A combined entity may be pushed earlier or positioned incorrectly because one or more sub-orders have dates that are being used as the basis instead of the lead order.

### Required behavior
Once orders are combined, the sequencing logic should not let sub-order avail dates override the combined entity’s canonical sequencing basis.

### Expected result
The combined entity should not jump forward/backward in sequence because of sub-order dates if the lead order is the intended canonical reference.

---

## 3. MAKE COMBINED ORDERS BEHAVE AS ONE UNIFIED SEQUENCING ENTITY

Please ensure that once orders are combined, they are treated as one unified sequencing entity in practice.

### Goal
The app should not sequence the combined order as if its members are still independent timing anchors.

### Expected result
The AI/resequencing logic should treat the combined group as one entity, using the lead order as the reference point.
