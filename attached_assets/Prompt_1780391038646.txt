Please enhance the **Auto-Sequence AI** so that its sequencing recommendations are more executable, more faithful to actual plant rules, and more transparent to the user.

The current AI already considers urgency, deadlines, changeovers, flexible advancement, profitability, and other sequencing signals. This request is to improve how the AI reasons and how its output is structured and applied.

---

# 1. MAKE THE AI SEQUENCE TRUE PRODUCTION ENTITIES, NOT JUST RAW ORDERS

Please revise the AI sequencing input/output model so that it sequences the actual **production entities** used by the scheduler.

### This should include entities such as:
- standalone orders
- combined groups
- generated/re-order entities
- linked source/generated production units where applicable

### Important
If combining happens before AI sequencing, then the AI should rank the **post-combination production entities**, not only the raw individual orders.

### Goal
Make the AI sequence the same units that the scheduler and preview/apply flow actually use.

### Expected result
The AI recommendation becomes more faithful to the final executable schedule.

---

# 2. USE STABLE BACKEND IDS / ENTITY IDS EVERYWHERE IN THE AI SEQUENCING FLOW

Please ensure all AI sequencing is tied to stable backend identifiers such as:
- `entity_id`
- `order_id`
- `production_group_id`

### Goal
Avoid ambiguity caused by:
- repeated product names
- repeated material codes
- unstable UI row order
- post-combination grouping

### Expected result
Every AI recommendation can be mapped cleanly from:
- input
- output
- preview
- apply-to-schedule
- transparency table
- history log

---

# 3. GIVE THE AI EXPLICIT DEPENDENCY CONSTRAINTS BETWEEN ENTITIES

Please improve the AI context so it understands entity dependencies, not just isolated rows.

### Examples of dependencies the AI should understand
- generated re-order must not be placed before source order
- source/generated entities may need proximity
- combined groups are indivisible
- some entities must remain after or before another entity
- some linked entities should not be separated beyond a practical threshold if such rules exist

### Goal
Reduce cases where the deterministic layer has to heavily correct the AI’s proposed sequence.

### Expected result
The AI’s suggested sequencing becomes more operationally valid from the start.

---

# 4. GIVE THE AI MORE EXPLICIT LINE-LEVEL AND DAY-LEVEL WORKLOAD CONTEXT

Please enrich the AI input with line/day workload shape, not only order-level fields.

### Examples of additional context
- total hours loaded per day on each line
- remaining daily capacity
- lightly loaded vs congested days
- front-loaded vs back-loaded schedule shape
- idle windows / underutilized days
- over-capacity or near-capacity days
- cumulative run hours up to each deadline window

### Goal
Help the AI make more realistic placement recommendations, not just relative ranking decisions.

### Expected result
The AI can better judge whether advancing or delaying an order actually improves the schedule.

---

# 5. GIVE THE AI A STRONGER NOTION OF DATE FEASIBILITY, NOT JUST RANKING

Please improve the AI output so it reasons separately about:
- rank/order in sequence
- suggested date
- placement feasibility

### Goal
Avoid situations where the AI gives a rank that later conflicts with date-based sorting or feasibility rules.

### Expected result
The AI output should better align ranking with actual date viability.

---

# 7. GIVE THE AI EXPLICIT CONTEXT ABOUT HARD CONSTRAINTS ENFORCED BY THE APP

Please make sure the AI prompt/context explicitly includes the real operational rules enforced by the app, including but not limited to:

- in-production orders cannot be moved
- critical / urgent / MTO orders may be pinned by deadline
- shutdown lines cannot receive orders
- line eligibility restrictions
- combined entities are indivisible
- generated/source dependencies
- chronology / avail-date constraints
- completion-feasibility constraints
- any other real refinement-layer rule currently used by the scheduler

### Goal
Help the AI propose strategies that are closer to what the app can actually apply.

### Expected result
AI Rank and Final Rank should diverge less often because the AI understands the true operating rules better.

### Goal
