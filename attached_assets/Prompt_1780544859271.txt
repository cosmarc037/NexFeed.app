## 1. REMOVE THE `+14 DAYS` LATE FLEXIBILITY FOR SUFFICIENT ORDERS

Right now, Sufficient orders can still be scheduled later, up to `+14 days`.

Please remove that behavior.

### Required change
Do not allow Sufficient orders to be pushed beyond their target/avail-date logic using the current `+14 day` flexibility.

### Goal
Make the schedule align with the actual Future Dispatches planning horizon instead of extending beyond it.

### Expected result
Sufficient orders should no longer be scheduled using the old +14-day late-production window.

---

## 2. ALIGN SUFFICIENT-ORDER SCHEDULING WITH THE 10-DAY FUTURE DISPATCHES WINDOW

The client’s planning basis is the **Future Dispatches** data, which is:
- updated daily
- based on the next 10 days of demand
- and used to determine whether a material is:
  - Critical
  - Urgent
  - Monitor
  - Sufficient

Please make the scheduling logic align to that same 10-day planning window.

### Goal
Keep the sequencing and date behavior consistent with the same rolling demand horizon used to compute order status.

### Expected result
Scheduling decisions for Sufficient orders should no longer rely on a horizon that is longer than the Future Dispatches planning window.

## 3. KEEP FUTURE DISPATCHES AS THE BASIS FOR ORDER STATUS

- Inventory
- daily forecast columns

and statuses such as:
- Critical
- Urgent
- Monitor
- Sufficient

### Goal
The app should continue using this daily-updated status logic as the operational basis for prioritization.

### Expected result

## 4. ALLOW THE AI TO ADVANCE ORDERS EARLIER IF THAT MAKES SENSE, BUT NOT PUSH SUFFICIENT ORDERS LATER BEYOND THE TARGET WINDOW

### Required behavior

- delay / produce Sufficient orders later using the previous +14-day allowance

### Goal
Keep early flexibility when useful, but remove the excessive late drift.

### Expected result
The AI can still optimize early production, but not by extending Sufficient orders beyond the intended planning horizon.
